Media Writing Skill Architecture

一个分步确认、逐步落盘的写作工作流

这个 skill 不是一次性吐出成稿的写作器，而是一台交互式内容生产流水线。它先判断任务类型，再锁定简报、骨架、叙事、说服、传播五层中间件，最后才整合主稿与平台改编。每推进一步，都要先提问、等待确认、写文件、再进入下一步。

核心设计

把“写作”拆成可检查、可回退、可复用的中间产物链，而不是把所有判断埋进一次回复里。

目录固定写入 posts/YYMMDD/{slug}/，从会话记录到最终校验形成完整痕迹。

Canonical Steps 9 从 Step 0 初始化到 Step 8 最终校验。
Artifact Files 8+ 默认至少生成 00 到 08 这一组过程文件。
Task Routes 6 从零写作、重写优化、观点文、个人表达、传播优化、多平台改编。
Core Layers 4 结构、叙事、说服、传播四层同时覆盖。
Config Scopes 2 项目级与用户级 EXTEND.md 共同提供默认值和扩展骨架。
01

运行模型

系统的主干不是“写文章”，而是“做决策”。先识别输入来自什么任务，再把写作拆进九个步骤。每一步都只推进一个关键判断，防止在信息不充分时直接产出失控成稿。

Input Surface
输入不是统一的
它可能是一个主题、一份旧稿、一段口播转文字、研究笔记、失败文案，或者仅仅是一句想表达的话。
主题 草稿 笔记 案例
Routing
先判断任务类型
任务路由决定要读哪些参考资料、要补哪类中间件，以及该优先诊断还是优先生成。
Control Loop
一轮只推进一件事
每一步必须遵守“提问或给待确认内容 → 等待用户回复 → 写文件 → 请求确认”的固定节奏。
如果用户试图跳步，skill 会先补齐缺失中间件再继续；如果用户已经给了完整草稿，也不会跳过流程，只是把更多精力放在诊断和重组，而不是重新收集素材。
Route A
从零写作
重点加载结构、叙事、传播三类参考，先补最小简报，再选骨架。
Route B
重写优化 / 观点文
先诊断结构与说服链条，再决定是否重写与如何重写。
Route C
个人表达 / 多平台改编
前者强化叙事细节，后者保留核心判断但改变节奏、顺序与表达形态。
Step 0
初始化会话
识别任务、读取扩展配置、创建 posts 目录、写 00-会话。
Step 1
锁定简报
确认平台、读者、核心判断、素材与目标动作。
Step 2
确定骨架
给出 1 到 3 个候选结构，并等待用户选择或授权代选。
Step 3
补叙事张力
补场景、冲突、变化、细节，不让正文变成说明书。
Step 4
建说服链
把证据、机制、反方、边界至少补齐三项。
Step 5
设计传播点
独立产出标题、钩子、金句与转发动机。
Step 6
整合主稿
基于已确认文件整合成文，不允许绕过前置确认。
Step 7
平台改编
保持信息不变，只调整平台节奏、结构和表达形式。
Step 8
最终校验
检查开头、场景、机制、标题匹配度与用户最终确认状态。
02

能力栈

写作被建模成四层叠加。架构上的关键点在于：这四层不是并列的装饰项，而是后续每一步都要显式落盘、显式确认的中间件。

Layer 1
结构层
负责结论先行、分组顺序、段落边界与章节骨架，解决“写清楚”。
结论 分组 顺序
Layer 2
叙事层
负责张力、场景、人物、节奏与细节，解决“写好看”。
场景 冲突 变化
Layer 3
说服层
负责证据、机制、反方与边界，解决“写可信”。
证据 机制 边界
Layer 4
传播层
负责标题、钩子、金句与转发动机，解决“写得出去”。
标题 钩子 转发
Why This Matters
它避免两类常见失败
一类是只修结构，最后像说明书；另一类是只追求标题和情绪，正文却没有支撑。四层拆分让“好看”和“可信”都能被显式检查。
Default Skeletons
骨架库决定了正文主形状
内置五种主骨架：结论-原因-证据-动作、问题-误区-重构-方法、现象-机制-推演-决策、故事-冲突-反转-提炼、清单-分类-示例-边界。
03

文件链路

真正让这个 skill 可复盘的，不是提示词，而是它强制把每个决策沉淀成文件。用户可以在同一主题目录下看到整个生成链，而不是只拿到一篇无法解释来源的主稿。

Session Core
初始化与简报
  • 00-会话.md 记录任务类型、平台、扩展配置、待确认项与下一步。
  • 01-简报.md 区分用户明确提供与模型推断待确认内容。
Content Middlewares
骨架到说服
  • 02-骨架.md 候选骨架、推荐理由、章节顺序、小结论。
  • 03-叙事.md 候选开场场景、核心冲突、关键变化、细节。
  • 04-说服.md 主张、证据、机制、反方、边界、风险点。
Distribution
传播与改编
  • 05-传播.md 标题、钩子、金句、转发动机。
  • 07-改编.md 同一判断在公众号、小红书、口播、短帖中的形态变化。
Final Delivery
主稿与收尾
  • 06-主稿.md 显式注明基于哪些已确认文件整合而成。
  • 08-校验.md 列出通过项、未通过项、建议修改与完成状态。
Storage Contract
目录规则就是状态机
所有文件写入 posts/YYMMDD/{slug}/。命名冲突时追加 -v2、-v3，而不是覆盖旧决策。也就是说，文件系统本身承担了状态追踪与版本保护功能。
Rewrite Variant
重写任务走另一条最小链
当输入是现有草稿时，至少要生成 00-会话、01-诊断、02-重写计划、03-主稿。它不是删减版，而是把“先诊断再重写”变成更短的专用路径。
04

依赖与扩展

这个 skill 把“知识来源”和“执行模板”分开。参考资料负责原则与判断框架，模板负责落盘结构，EXTEND.md 负责项目级或个人级偏好注入。

References
按任务路由读取的知识层
  • structure-and-argument.md 决定骨架选择与论证链条。
  • narrative-and-expression.md 给叙事抓手、表达节奏与细节策略。
  • persuasion-and-trust.md 补说服结构，避免只有立场没有支撑。
  • titles-and-distribution.md 约束标题和传播机制，不靠模板硬套。
  • workflow-templates.md 提供最小简报与重写诊断清单。
  • llm-prompts.md 在需要稳定格式时提供复用提示词模板。
Templates
按步骤落盘的结构层
  • 00-会话-template.md 初始化与进度可视化。
  • 01-简报 / 01-诊断 写作简报与草稿诊断双入口。
  • 02-骨架 / 02-重写计划 正常路径与重写路径双模板。
  • 03-叙事 到 08-校验 把五层中间件和最终 QA 变成固定文件结构。
EXTEND.md
外部偏好注入层
  • 项目级 .daqi-skills/media-writing/EXTEND.md
  • 用户级 $HOME/.daqi-skills/media-writing/EXTEND.md
  • 注入内容 默认平台、读者、风格、目标动作、全局规则、扩展骨架。
  • 优先级 用户当轮要求 > 会话确认内容 > 项目扩展 > 用户扩展 > 内置默认值。
05

约束规则

这些规则决定了它为什么更像“工作流系统”而不是“写作提示词”。从交互模式到文件命名，再到诊断格式，所有约束都在压缩随机性，提升可解释性与可复用性。

Interaction Guardrails
不允许一步出整稿
  • 只要用户给的是模糊需求，就必须按回合制推进。
  • 每一步都要先给摘要，再请求确认后继续。
  • 用户跳步时，先补缺失中间件，而不是迎合跳步。
Storage Guardrails
不覆盖、只增量
  • 禁止覆盖已有文件，除非用户明确要求覆盖。
  • 冲突命名通过 -v2、-v3 递增解决。
  • slug 压缩主题但保持语义，不为避重而随意改题。
Reasoning Guardrails
可以推断，但不能编造
  • 信息不足时可以补最小简报，但推断项必须标成待确认。
  • 可留白、可写待补充，不能伪造经历、数据或案例。
  • 诊断必须基于全文内容，而不是词频式套路判断。
读图建议：如果你要继续扩这个 skill，最值得优先看的不是主稿模板，而是 Step 0 的初始化约束、Step 2 的骨架选择逻辑，以及 EXTEND.md 的优先级设计。这三处决定了它能否从“单次生成器”稳定升级成“可维护的写作操作系统”。
