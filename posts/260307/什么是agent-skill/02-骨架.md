# 02 骨架

## 候选骨架

## 扩展骨架命中情况

- 是否读取到 EXTEND：是
- 命中的扩展骨架：技能介绍型、经验分享型
- 命中原因：主题是介绍一个自己正在用的 skill；同时目标不是抽象定义，而是通过真实案例和演示把人讲懂

### 方案 A

- 骨架类型：轻叙事引入 -> 真实演示 -> 结构拆解 -> 定义收束
- 适用理由：最符合你确认的主路径。先用 learning-loop 跑一次，让听众先看到“agent skill 真会干活”；再回头解释为什么它像 APP、为什么离不开 agent、SKILL.md 和目录结构各自负责什么；最后收束成一个通用理解。这个结构对口播尤其有效，因为它先给画面和动作，再给抽象概念。
- 风险点：如果演示讲太细，后半段的结构总结会被挤压，听众容易记住 learning-loop，却没提炼出 agent skill 的共性。

### 方案 B

- 骨架类型：问题 -> 误区 -> 重构 -> 演示 -> 总结
- 适用理由：适合先打掉大众对 skill 的误解，比如把它当提示词、模板或一段说明书。然后给出“它更像一个挂在 agent 上的应用”这个新框架，再用 learning-loop 证明。
- 风险点：前面概念铺垫会偏多，不如方案 A 直接、有画面，可能削弱你要的“先把人带进去”的效果。

### 方案 C

- 骨架类型：故事 -> 冲突 -> 反转 -> 拆解 -> 提炼
- 适用理由：如果你想强化“很多人听过 skill 还是不懂，我也卡过，后来我做了一个 learning-loop 才讲明白”这种真人感，这个结构有张力。
- 风险点：故事一多就会稀释教学密度，不一定适合已经在用 agent 的创作者，他们更想尽快进入结构和方法。

## 推荐方案

- 推荐选择：方案 A
- 推荐理由：它同时满足你确认的三个目标。第一，口播稿需要先让人看到动作；第二，读者是已经在用 agent 的创作者，他们更吃“先看能不能跑，再看怎么构成”；第三，结尾目标是“彻底听懂 agent skill 是什么”，那就最适合从一个真实可运行案例倒推共性定义。
- 如果用户不选择时的默认执行方案：方案 A
- 是否来自扩展骨架：是，属于“技能介绍型”和“经验分享型”的混合变体

## 章节顺序

### 1. 开头

- 这一节要解决什么：先把听众从抽象定义里拉出来，告诉他这次不空讲概念，而是直接拿一个我最近真的在用的 learning-loop 给他看
- 小结论：skill 这件事，光听别人讲定义很难懂，得先看它怎么干活

### 2. 主体一

- 这一节要解决什么：先演示 learning-loop 怎么被调用、怎么进入学习流程、怎么读写文件、怎么形成一套实际运行的工作流
- 小结论：所谓 skill，不是一段好听的话，而是一套能被 agent 调起并实际执行的能力包

### 3. 主体二

- 这一节要解决什么：从刚才的运行结果往回拆，解释 SKILL.md 负责什么，名称与描述为什么重要，触发词、语法、输入输出、工作流程、记忆系统分别扮演什么角色
- 小结论：agent skill 之所以像 APP，不是因为它有一个文件夹，而是因为它有入口、说明、流程、资源和执行边界

### 4. 主体三

- 这一节要解决什么：继续拆 learning-loop 的目录结构，解释 assets、references、scripts 为什么分开，以及 skill 可以怎么调用其他 skill、tools、插件、MCP、联网和脚本
- 小结论：一个 skill 的本质，是把“什么时候触发、调什么工具、按什么顺序做、结果写到哪里”封装成 agent 可复用的工作单元

### 5. 结尾

- 这一节要解决什么：把 learning-loop 这个具体例子收束成一句通用定义，帮听众彻底听懂“为什么必须叫 agent skill”
- 小结论：脱离 agent，skill 只是说明书；挂到 agent 上，skill 才变成真正会工作的应用
