# 04 说服

## 主张

- 核心主张：skill 应该叫 agent skill。它像一个应用、一个软件、一个 APP，但前提是它必须挂在 agent 上，才能被触发、调工具、走流程、读写文件、产出结果。
- 这条主张想改变读者什么认知：把读者从“skill 不就是一段提示词 / 一个模板 / 一个目录”这类静态理解，拉到“skill 是 agent 可调用的能力封装”这类动态理解。

## 证据

- 经验案例：learning-loop 不是只写了一份说明，而是明确规定了命令语法、执行流程、写入路径和记忆更新方式。用户输入学习主题后，它会初始化 session、创建目录与脑图、执行诊断到测评的四步循环，并把结果落到不同文件里。
- 数据或事实：learning-loop 在 SKILL.md 里给了明确命令格式 `/learning-loop topic="主题名" [slug="custom-slug"] [cycle=N] [action=resume|status]`；定义了 Step 1 到 Step 4 的执行链；还明确写了记忆文件结构，比如 `meta.md`、`journal.md`、`cycles/cycleN/diagnosis.md`、`curriculum.md`、`quiz.md`、`assessment.md`。
- 对比样本：如果只把 learning-loop 的目录或一段介绍文案单独拿出来，它最多只是一个说明包；但一旦它被 agent 按描述触发，去读 references、调 scripts、写 session 文件，它就变成了一个真正会工作的能力单元。这个差别，正是“文档”和“agent skill”的差别。
- 当前证据缺口：这篇暂时不做 live demo 截图或逐步命令回放，所以说服主要依赖结构证据和真实案例拆解，而不是现场运行记录。

## 机制

- 为什么会这样：因为 agent skill 的核心不是“写了什么文字”，而是“agent 看到什么描述会触发、触发后按什么语法接收输入、再按照什么工作流去调用工具和资源”。
- 关键变量：触发入口；输入语法；工作流步骤；可调用资源；记忆读写路径；输出落点。
- 变量之间如何作用：description 决定什么时候被用；语法决定怎么接收任务；工作流决定执行顺序；references、assets、scripts 提供执行所需的外部支撑；记忆系统保证过程有状态可延续。只有这些变量被 agent 串起来，skill 才不是静态说明，而是动态应用。

## 反方与异议

- 常见反对意见：skill 不就是一份写得更复杂的提示词，或者一个文件夹模板吗？
- 可能的反例：有些 skill 看起来也只有一个 SKILL.md，没有复杂脚本、没有大量资源文件。
- 如何回应：判断一个东西是不是 skill，不看它文件多不多，而看它是否真的定义了触发条件、任务边界和执行路径。一个轻量 skill 可以很小，但只要它能让 agent 在特定场景下稳定接管任务、调用必要工具并输出结果，它仍然是 skill。反过来，一个目录再完整，如果 agent 根本不会触发、不会执行，那也只是静态资料。

## 边界与代价

- 适用条件：这个“skill 像 APP”的类比，适合用来帮助已经在用 agent 的创作者快速建立心智模型，尤其适合解释工作流型、工具调用型、带记忆与文件输出的 skill。
- 不适用条件：如果对方完全没接触过 agent，只讲“像 APP”可能还不够，他可能还需要先理解 agent、本地工具、MCP、文件系统这些基础概念。
- 代价或副作用：APP 这个类比虽然好懂，但也容易让人误以为 skill 一定要很复杂、一定要有很多文件。这里必须补一句，核心不是规模，而是是否能被 agent 触发并执行。

## 风险提醒

- 哪些判断不能说太满：不能把所有 skill 都说成完整软件，也不能把所有提示词都排除在外。更准确的说法是，skill 是一种围绕 agent 执行而组织起来的能力封装。
- 哪些地方还需要补证：如果后面你希望更强的实操感，可以在主稿里补一个更具体的“运行瞬间”，比如输入主题后如何创建 session 目录、如何进入 cycle1、如何把 assessment 写回文件。

## 待用户确认

- 哪条边界必须写清：推荐写清“不是文件多就叫 skill，关键是能否被 agent 触发并执行”
