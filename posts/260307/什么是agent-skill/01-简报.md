# 01 简报

## 用户明确提供

- 主题：skill 到底是什么
- 平台：口播稿
- 目标读者：已经在用 agent、准备自己写 skill 的创作者
- 核心判断：skill 更像一个应用、一个软件、一个 APP，但它必须叫 agent skill，因为脱离 agent 它根本跑不起来
- 目标动作：先把 skill 听懂，再愿意照着案例去写自己的 skill

## 模型推断待确认

- 作者身份：以“最近真的做了一个 learning-loop 的实操者”身份来讲，而不是纯概念解释者（已确认）
- 期望语气：口语化、讲解型、带一点“很多人讲过但你还是没听懂，我换个方式给你讲明白”的推进感（沿用项目默认风格）
- 读完后希望获得的结果：听众不仅理解什么是 skill，还能知道一个 skill 至少由哪些部分构成、为什么必须挂在 agent 上、怎么通过一个真实案例判断它是不是能用（已确认结尾优先落在“彻底听懂 agent skill 是什么”）

## 可用素材

- 已有经历：最近做了一个交互式学习 skill，可以作为完整演示案例
- 已有案例：learning-loop，能具体展示 SKILL.md、语法、输入输出、工作流、记忆系统、assets、references、scripts
- 已有研究/笔记：.agents/skills/skill-creator/SKILL.md，可用于补充 skill 规范和结构解释
- 已有旧稿：posts/260307/学习skill介绍/learning-loop-overview.html 可参考其中的讲解方式或视觉表达

## 风险与空缺

- 仍然缺少的关键信息：骨架是否接受“先演示，再拆结构，再收束定义”的顺序；是否要在演示里显式说出 trigger/description 的关系
- 可能导致后续偏差的地方：如果演示跑太长，会挤压结构抽象总结；如果字段拆解太轻，听众会理解案例但不一定抽象出 agent skill 的共性

## 建议写作方向

- 推荐文章类型：技能介绍型 + 经验分享型混合
- 推荐主承诺：我不用抽象定义讲 skill，我直接拿一个真实 skill 拆给你看，你看完就知道 agent skill 到底是什么、为什么它像 APP、以及一个 skill 最少是怎么组织起来的
- 推荐下一步骨架方向：轻叙事引入 -> 先打破“skill 很玄”的误解 -> 直接演示 learning-loop 怎么被调用和运行 -> 从运行结果倒推 skill 的结构与字段 -> 收束成 agent skill 的通用定义

## 待用户确认

- 确认项 1：骨架是否确认采用“先演示，再拆结构，再收束定义”的顺序？
- 确认项 2：是否需要在主体里单独拔出一小节讲 trigger 和 description 的关系？
- 确认项 3：骨架确认后，是否直接进入叙事层？
