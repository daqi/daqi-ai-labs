# 栏目设计

## 一、设计原则

这套栏目设计，不是为了把账号做得很满，而是为了让内容有主轴、有层次、有复利。

栏目设计遵守四个原则：

1. 先围绕真实任务，不围绕空泛概念。
2. 先做能演示结果的内容，再做纯认知表达。
3. 先让用户感受到你有用，再逐步建立你有判断。
4. 每个栏目都要能沉淀案例、模板、工作流或课程素材。

## 二、账号总主轴

账号总主轴只有一句话：

用 AI 帮普通人把真实、重复、高频的任务，变成可执行的工作流。

在这个主轴之下，栏目不是分散的兴趣区，而是不同功能区：

- 有的栏目负责建立信任
- 有的栏目负责打强痛点
- 有的栏目负责强化你自己的业务身份
- 有的栏目负责建立结构判断力

## 三、当前阶段栏目结构

当前阶段不适合同时跑 4 个核心栏目。更适合的结构是：1 条主航道，2 条辅助航道，1 条低频信任层。

### 主航道：Agent Skill 实战

定位：把“做 skill”这件事本身做成内容主线。

目标用户：

- 已经在用 AI 或 Agent，但还不会把任务做成可复用工作流的人
- 对 skill、workflow、SOP、自动化实战感兴趣的人
- 想看真实案例，而不是只想听概念的人

内容方向：

- 今天做了一个什么 skill
- 这个 skill 解决了什么真实问题
- 这个 skill 是怎么跑的
- 这个 skill 为什么这么设计
- 做这个 skill 时踩了什么坑

内容形式：

- 产品演示型
- 案例实验型
- 拆解复盘型

栏目作用：

- 建立你的核心身份：做 skill 的人
- 把 skill 生产过程直接变成内容资产
- 为后续产品、服务、咨询和方法论沉淀做入口

更新建议：

- 每周 4 到 5 条，优先保证

### 辅助航道一：Skill 拆解与经验总结

定位：把零散实战上升为可复用判断。

目标用户：

- 已经想做 skill，但还没有形成稳定方法的人
- 想少踩坑、少走弯路的人

内容方向：

- skill 不是 prompt，而是 SOP
- 为什么不要一上来就生成 skill
- skill 的状态管理、扩展机制和底座能力
- 这周我做 skill 得到的 3 个经验

内容形式：

- 判断总结型
- 方法拆解型
- 连载专题型

栏目作用：

- 建立结构判断力
- 沉淀方法论资产
- 把单次实战变成可复利的内容模块

更新建议：

- 每周 1 到 2 条

### 辅助航道二：基建与研究雷达

定位：把信源抓取、监控系统、vibe coding 小工具等基建，作为主航道的发动机，而不是独立业务线。

目标用户：

- 对 AI 前沿动态、信息抓取、研究工作流感兴趣的人
- 想知道你背后怎么找题、怎么搭基建的人

内容方向：

- 怎么抓 X 上关于 agent、skill、openclaw 的讨论
- 我怎么用一个小工具提升 skill 生产效率
- 这个基建为什么值得做，为什么不值得做

内容形式：

- 小实验型
- 工具演示型
- 工作台展示型

栏目作用：

- 为主航道持续供给题材和信息源
- 展示你的研究和生产底座
- 把基建成果转化成可发内容，但不喧宾夺主

更新建议：

- 每周 1 条

### 低频信任层：自我探索与栏目判断

定位：回答“你为什么做这件事”“你在建立什么长期结构”。

目标用户：

- 愿意长期看你的人
- 不只关心工具，还关心判断和主线的人

内容方向：

- 自我探索
- 栏目设计
- 为什么我决定边做基建边发内容
- 我对 AI、自媒体、复杂系统和认知表达的长期判断

内容形式：

- 长文判断型
- 阶段复盘型
- 价值观表达型

栏目作用：

- 提高辨识度
- 建立长期信任
- 给主航道提供更深的叙事背景

更新建议：

- 每周 0 到 1 条，低频但持续

## 四、当前阶段分工

这几个栏目不是并列竞争关系，而是层级关系。

主航道负责：

- 建立你的核心身份
- 直接承接增长和转化
- 持续制造真实内容素材

辅助航道负责：

- 把主航道拆清楚、讲透、放大
- 用基建和研究系统为主航道供给弹药

低频信任层负责：

- 建立长期判断
- 解释你的主线为什么成立
- 防止账号只剩工具演示，没有人格和结构

## 五、当前阶段内容比例建议

当前阶段建议按下面比例布局：

1. Agent Skill 实战：60%
2. Skill 拆解与经验总结：20%
3. 基建与研究雷达：10%
4. 自我探索与栏目判断：10%

这样分配的原因是：

- 现在最该建立的是“做 skill 的人”这个身份
- 经验总结能把单条内容变成方法资产
- 基建内容要做，但应该服务主航道，不该抢时间
- 自我探索和栏目判断很重要，但当前不该高频

## 六、每个栏目的标准结构

为了让账号风格统一，所有栏目优先采用同一条底层结构：

1. 开头先展示结果
2. 中间快速讲清架构
3. 现场演示一个关键动作
4. 结尾抛互动问题

也就是说，不同栏目讲的是不同任务，但展示逻辑尽量统一。

这样做的好处是：

- 用户更容易形成记忆
- 你自己更容易稳定产出
- 每条内容都更容易从传播走向转化

## 七、栏目命名建议

为了让栏目更容易记忆和连续更新，建议统一用这种命名方式：

- AI 学习实验室
- AI 求职实验室
- AI 内容实验室
- AI 判断室
- 本周 AI 小实验

如果你想更口语一点，也可以用：

- 学习这件事，我拿 AI 做了个实验
- 求职这件事，我拿 AI 做了个实验
- 做内容这件事，我拿 AI 做了个实验
- 这个 AI 工作流，到底值不值得学

## 八、当前最优先启动顺序

如果只从执行效率、真实资产和增长目标出发，建议启动顺序如下：

1. Agent Skill 实战
2. Skill 拆解与经验总结
3. 基建与研究雷达
4. 自我探索与栏目判断

原因很简单：

- 你手里最真实、最强的资产就是已经做出来的 skill
- skill 实战最容易建立第一信任和账号记忆
- 拆解与经验总结能把实战变成可复利资产
- 基建要做，但它的职责是给主航道供给，不是自成一摊
- 自我探索重要，但更适合低频拉深，不适合当前阶段高频主打

## 九、判断这套栏目是否有效

这套栏目设计是否跑通，不先看粉丝量，先看三件事：

1. 你是否能稳定从主航道持续产出选题
2. 用户是否开始记住你是在做“skill 实战和工作流实验”的人
3. 内容是否开始沉淀成案例、模板、工作流和课程素材

如果这三件事开始出现，这套栏目设计就是成立的。

## 十、长期可扩展栏目库

上面是当前阶段的执行结构，不代表长期只能做这些。

等主航道跑稳之后，可以再把下面这些方向逐步扩成独立栏目：

- AI 学习实验室
- AI 求职与成长实验
- AI 自媒体生产实验
- AI 工作流判断室
- 本周 AI 小实验

也就是说，这些不是现在不做，而是现在先收进储备位，不和主航道抢执行力。

## 十一、可单独连载的专题：Agent Skills 实战系列

如果你准备把 Agent Skills 写成多篇实战内容，建议不要写成一篇大而全的方法论，而是拆成一个连续专题。

这个专题适合挂在 AI 工作流判断室 和 AI 自媒体生产实验 之间。

原因是它同时具备两种价值：

- 一方面，它能展示你对 AI 工作流的判断，不只是会用工具。
- 另一方面，它又有明确演示空间，能让人看到你怎么把一个想法做成可复用系统。

建议这个系列的每篇内容都固定用同一结构：

1. 这篇想分享的观点
2. 看完本篇你能解决什么问题
3. 一个简单实操演示
4. 最后给一个可直接套用的动作

下面是一个适合连续发布的 8 篇拆法。

### 第 1 篇：skill 不是 prompt，而是 SOP

分享的观点：

- 很多人做 skill，其实只是把 prompt 写长了。
- 但真正有价值的 skill，不是提示词堆砌，而是一个可复用、可执行、可迭代的工作流。

看完本篇能解决的问题：

- 理解 skill 和 prompt 的根本差别。
- 知道为什么很多“看起来很强”的 skill，实际复用率很低。
- 建立做 skill 时最基础的判断标准。

实操演示：

1. 先拿一个普通 prompt 当例子。
2. 再把它拆成输入、步骤、输出、异常处理四部分。
3. 对比说明：什么叫一次性回答，什么叫可复用工作流。
4. 最后给出一个最小 skill 骨架示意。

### 第 2 篇：为什么不要一上来就生成 skill

分享的观点：

- skill 失败，很多时候不是生成能力不够，而是前期问题没定义清楚。
- 先讨论方案，再生成 skill，质量会高很多。

看完本篇能解决的问题：

- 知道生成前到底要先想清楚什么。
- 避免把模糊需求直接丢给 Agent，最后得到一堆不能复用的产物。
- 学会用一套简单框架讨论 skill 方案。

实操演示：

1. 随便拿一个需求，比如“做一个写作 skill”。
2. 现场演示先不写 skill，只问目标用户、输入、输出、流程、边界。
3. 展示讨论前后的方案差异。
4. 总结一个生成前检查清单。

### 第 3 篇：做 skill 之前，先借人类的方法论

分享的观点：

- 做 skill 不能只问 AI“帮我写一个”。
- 更好的路径是先问这个领域有哪些书、理论、经典方法，再把它们压缩成 skill。

看完本篇能解决的问题：

- 知道为什么很多 skill 看起来能跑，但质量不稳定。
- 学会先调研方法论，再做流程设计。
- 理解“AI 不是替代思考，而是压缩知识调用成本”。

实操演示：

1. 选一个主题，比如学习、写作或面试。
2. 先让 AI 列相关书籍、理论和成熟方法。
3. 再把这些方法论压缩成 3 到 5 个关键步骤。
4. 最后展示怎么把步骤转成 skill 草图。

### 第 4 篇：怎样分析一个优秀 skill 的架构

分享的观点：

- 很多人会用 skill，但不会拆 skill。
- 真正拉开差距的能力，是看懂别人的结构，然后抽象出可复用部分。

看完本篇能解决的问题：

- 学会怎么看一个 skill 的结构，而不是只看表面文案。
- 知道分析 skill 时应该重点看什么。
- 能从优秀案例里提炼自己的设计方法。

实操演示：

1. 选一个现成 skill。
2. 用 visual-explainer 或手动方式拆它的输入、流程、状态、输出。
3. 找出它真正厉害的地方，是结构、交互还是状态设计。
4. 最后输出一张“可借鉴点”清单。

### 第 5 篇：一个高级 skill 需要哪些底座能力

分享的观点：

- skill 的上限，很多时候不在文案，而在底座。
- 没有浏览器、API、脚本、第三方包这些能力，很多 skill 只能停留在文本层。

看完本篇能解决的问题：

- 知道为什么有些 skill 看起来聪明，但做不出真实结果。
- 理解高级 skill 的底层能力组成。
- 知道自己下一步该补哪类底座。

实操演示：

1. 列一个纯文本 skill 能做什么。
2. 再列一个接浏览器或脚本后的 skill 能多做什么。
3. 用一个小例子展示 npx -y bun scripts/xxx.ts 这类脚本调用思路。
4. 最后总结“文本层 skill”和“执行层 skill”的差别。

### 第 6 篇：为什么交互式 skill 比一次性 skill 更强

分享的观点：

- 一次性 skill 解决的是局部效率。
- 交互式 skill 解决的是复杂任务里的持续逼近问题。

看完本篇能解决的问题：

- 理解什么时候该做一次性输出，什么时候该做多轮交互。
- 知道交互式 skill 为什么更适合学习、写作、咨询这类场景。
- 学会设计一个简单的多轮结构。

实操演示：

1. 用同一个任务，分别演示一次性输出和多轮交互输出。
2. 对比两种方式在准确度、参与感、可调整性上的差异。
3. 给出一个最小多轮流程：提问、确认、输出、迭代。
4. 说明为什么人和 skill 会在这个过程中一起变强。

### 第 7 篇：有工作流的 skill，为什么一定要有进度文件

分享的观点：

- 只要是多步骤任务，没有状态管理，后面一定会乱。
- 进度文件不是附属品，而是工作流 skill 的核心部件。

看完本篇能解决的问题：

- 理解为什么很多工作流 skill 用一次还行，第二次就乱。
- 学会给 skill 增加中断、恢复、回溯能力。
- 建立“过程资产要落盘”的工程意识。

实操演示：

1. 先展示一个没有进度文件的多步骤流程会出现什么问题。
2. 再展示有进度文件时，每一步结果如何被保存。
3. 演示中断后如何继续。
4. 最后总结一个最小状态管理结构。

### 第 8 篇：我现在做 skill 的完整流程

分享的观点：

- skill 不是灵感驱动，而是流程驱动。
- 一旦流程稳定，skill 的成功率和复用率都会明显提升。

看完本篇能解决的问题：

- 获得一套完整可照抄的 skill 生产流程。
- 知道从想法到成品中间有哪些关键环节。
- 知道怎么把 skill 从“能跑”打磨成“能复用”。

实操演示：

1. 从一个真实需求开始。
2. 先调研方法论和案例。
3. 再讨论方案，明确输入输出和异常情况。
4. 用 skill-creator 生成第一版。
5. 补扩展机制、底座能力和状态管理。
6. 在真实使用里迭代修正。

## 十一、这个系列的统一写法

为了让这个系列更像一个栏目，而不是零散笔记，建议每篇都尽量统一写法。

建议结构：

1. 开头先抛一个反常识观点
2. 接着说清楚这篇解决什么问题
3. 中间演示一个最小流程
4. 结尾给一个可以马上照做的动作

开头可以多用这种句式：

- 很多人以为……但真正的问题是……
- 大多数人卡住，不是因为不会写，而是因为没先拆清……
- 这个东西之所以难做，不是因为工具不够强，而是因为……

## 十二、建议的发布顺序

如果你想让用户的理解层层递进，建议按这个顺序发：

1. 第 1 篇，先纠正认知
2. 第 2 篇，讲为什么先讨论方案
3. 第 3 篇，讲方法论来源
4. 第 4 篇，讲怎么拆优秀案例
5. 第 5 篇，讲高级 skill 的底座
6. 第 6 篇，讲交互式设计
7. 第 7 篇，讲状态管理
8. 第 8 篇，最后给完整流程

这样用户会从“知道你在讲什么”，慢慢走到“知道你真的做过，而且知道怎么做”。
