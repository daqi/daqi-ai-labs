# Agent Skills 实战经验

## 一句话定义

skill 不是一段 prompt，也不是一次性生成物。

skill 的本质是可复用、可迭代、可追踪的 SOP。它把一个领域里的做事方法，沉淀成可以稳定调用、持续进化的人机协作系统。

## 一、先把 skill 看对

1. skill 的本质是可复用 SOP。
2. 一个好 skill，不只是帮人省力，还要补足人的盲区、短板和认知缺口。
3. AI 不只是杠杆，它还压缩了人类知识的调用成本。做 skill 时，不应该只让它“执行”，还要让它“吸收前人的智慧后再执行”。

## 二、先讨论方案，再开始生成

1. 不要一上来就生成 skill，先把目标、边界、输入、输出、流程、异常情况聊清楚。
2. 可以先在网页或对话里把方案讨论透，再放到 Agent 里正式生成。
3. skill-creator 适合在方案已经相对清晰之后使用，而不是替代前期思考。

## 三、先借鉴人类智慧，再沉淀为 skill

1. 做一个 skill 前，先问 AI 这个主题相关的书籍、理论、经典方法和已有实践。
2. 不要直接从“我要一个功能”出发，而要先问“这个领域里最成熟的方法论是什么”。
3. 先把人类长期积累的方法论吸收进来，再把它们压缩成 skill，质量会高很多。
4. 使用 visual-explainer 分析优秀架构、已有工作流和前人经验，能更快看清结构，而不是只看表面写法。

## 四、skill 不是一次性交付，而是可扩展系统

1. skill 最好预留扩展入口，例如先加载 EXTEND.md，用来承接后续补充规则、特例、经验和领域变体。
2. 高级 skill 需要底座能力，而不只是文字组织能力。
3. 常见底座包括：浏览器操作、API 集成、第三方包调用、本地脚本执行。
4. 使用 npx -y bun scripts/xxx.ts 这类方式，可以在很多场景下简化依赖管理和脚本执行成本。

## 五、交互式 skill 才会持续变强

1. 交互式 skill 不只是“替你做”，还可以“和你一起做”。
2. 它可以实现交互式生产，也可以实现交互式学习。
3. 在这种模式下，人在提升，skill 也在提升，最后形成双向迭代。
4. 所以 skill 设计时，不能只考虑一次运行成功，还要考虑如何在多轮互动中不断逼近更好的结果。

## 六、有工作流的 skill，必须有状态管理

1. 只要 skill 有多步骤工作流，就应该有进度文件。
2. 进度文件的意义，不只是“记录做到哪了”，而是保证任务可以中断、恢复、回溯、审计。
3. 每一步都应该尽量保存阶段结果，这样后续才能链路追踪，也方便定位问题。
4. 没有状态管理的工作流 skill，短期能跑，长期很难维护，也很难迭代。

## 七、一个更稳的 skill 生产流程

1. 明确要解决的真实问题，而不是直接定义功能。
2. 先调研这个领域的书籍、理论、成熟 SOP 和优秀案例。
3. 用 visual-explainer 或其他方式拆清核心结构。
4. 先对话讨论方案，明确输入、输出、流程、状态和异常处理。
5. 再用 skill-creator 生成第一版。
6. 为 skill 补齐扩展机制、外部能力和状态管理。
7. 在真实使用中持续修正，逐步把 skill 从“能用”打磨到“可复用”。

## 八、当前结论

1. skill 的上限，不取决于 prompt 写得多花，而取决于你是否真的抽象出了一个可复用的工作流。
2. 好的 skill，不是一次生成出来的，而是在讨论、验证、使用、复盘中长出来的。
3. 真正高级的 skill，一定同时具备三件事：方法论基础、工程化底座、可持续进化机制。
