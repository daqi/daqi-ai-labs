# 自媒体写作

用这套工作流把零散想法、研究笔记、已有草稿或失败文案，转成结构清楚、叙事顺滑、论证可信、具备传播潜力的自媒体内容。

默认按“先判断任务类型，再进入交互式确认流程，逐步补足四层能力，最后做传播检修”的顺序执行，不要只做表面润色。

## 适用边界

- 这个 skill 负责完整写作链路：从零写、重写优化、结构重组、多平台改编、整稿交付。
- 如果用户明确要解决的是“小红书口播留人、开头钩子、掉线段、念不顺、结尾承接、连载预告”，优先使用 `xhs-doctor`，不要强行继续走完整写作流程。
- 如果用户先用本 skill 产出了主稿，后续再做小红书口播专项精修，也应切换给 `xhs-doctor`。

## 交互模式

这个 skill 必须以交互式方式运行，不允许在用户只给出一个模糊需求时直接一次性吐出整篇成稿。

执行时遵守以下硬约束：

1. 采用“你问我答”的回合制流程，一次只推进一个步骤。
2. 每个步骤都先提问或给出待确认内容，等待用户回复或确认后再进入下一步。
3. 每个步骤结束后都必须生成一个文件，落盘到 `posts/YYMMDD/{slug}/` 下。
4. 如果目标文件名已存在，绝不覆盖，改为追加版本号 `-v{n}`，例如 `01-简报-v2.md`。
5. 每一步都先给用户看本步产出摘要，再明确请求确认，例如“确认后我继续下一步”。
6. 如果用户跳步，先说明会补齐缺失中间件，再继续执行。
7. 如果用户已经给出完整草稿，也仍然按步骤推进，但可以在单步中少问、更多使用已有内容。

## 文件输出规则

所有中间产物和终稿都写入 `posts/YYMMDD/{slug}/`。

- `YYMMDD` 使用当前本地日期，例如 `260306`。
- `slug` 使用主题的短横线形式，优先简洁、可读、可复用；可保留中文，不要有空格。
- 首次进入流程时，如果目录不存在则创建。
- 文件名使用有序前缀，保证同一主题下能看到完整生成链路。
- 命名冲突时，按顺序尝试 `-v2`、`-v3`、`-v4`，直到找到可用文件名。
- 禁止覆盖用户已存在的文件，除非用户明确要求“覆盖”或“重写原文件”。

### Slug 规则

生成 `slug` 时使用以下优先级，不要随意命名：

1. 优先提取主题中的核心名词和核心判断，压缩成 3 到 8 个词。
2. 删除空格，统一改为短横线连接。
3. 删除无信息词，例如“关于”“如何”“为什么我们要”等空泛前缀，保留真正有辨识度的部分。
4. 中文主题可直接保留中文并用短横线连接，不强制转拼音。
5. 英文或中英混合主题统一转小写。
6. 删除路径不安全字符，例如 `/` `\` `:` `*` `?` `"` `<` `>` `|`。
7. 如果用户已明确给出目录名或 slug，优先使用用户指定值。
8. 如果主题过长，优先保留“对象 + 冲突/判断”，不要把整句标题原样拿来做目录名。

示例：

- `为什么做得越好越容易被换？` -> `为什么做得越好越容易被换`
- `AI 写作为什么越来越像流水线` -> `ai写作越来越像流水线`
- `普通人做个人 IP 的三个误区` -> `个人ip三个误区`

如果根据规则得到的 `slug` 与已有目录重名，不修改目录语义，继续在文件名层面追加版本号，而不是为同一主题强行改一个新目录名。

默认文件清单如下，可按任务类型裁剪，但每推进一步都必须至少生成一个文件：

- `00-会话.md`：记录任务类型、主题、日期、slug、平台、当前进度。
- `01-简报.md`：写作简报。
- `02-骨架.md`：文章骨架与章节顺序。
- `03-叙事.md`：叙事抓手、场景、冲突、细节。
- `04-说服.md`：证据、机制、反方、边界。
- `05-传播.md`：标题、钩子、金句、转发动机。
- `06-主稿.md`：整合后的主稿。
- `07-改编.md`：平台改编稿。
- `08-校验.md`：最终检查与修改决议。

优先复用以下模板生成对应文件，不要每次从零发明结构：

- `assets/session-templates/00-会话-template.md`
- `assets/session-templates/01-诊断-template.md`
- `assets/session-templates/01-简报-template.md`
- `assets/session-templates/02-重写计划-template.md`
- `assets/session-templates/02-骨架-template.md`
- `assets/session-templates/03-叙事-template.md`
- `assets/session-templates/04-说服-template.md`
- `assets/session-templates/05-传播-template.md`
- `assets/session-templates/06-主稿-template.md`
- `assets/session-templates/07-改编-template.md`
- `assets/session-templates/08-校验-template.md`

如果任务是“重写优化”或“草稿诊断”，也不能跳过落盘。至少要生成：

- `00-会话.md`
- `01-诊断.md`
- `02-重写计划.md`
- `03-主稿.md`
- 后续按需要继续生成传播和改编文件。

其中：

- `01-诊断.md` 优先使用 `assets/session-templates/01-诊断-template.md`
- `02-重写计划.md` 优先使用 `assets/session-templates/02-重写计划-template.md`

## 扩展配置

这个 skill 支持通过 `EXTEND.md` 做项目级和用户级扩展配置，用来补充默认骨架、默认平台、默认风格、默认目标读者、默认目标动作、全局写作规则，以及项目自定义骨架。

读取路径如下：

- 项目级：`.daqi-skills/media-writing/EXTEND.md`
- 用户级：`$HOME/.daqi-skills/media-writing/EXTEND.md`

优先级如下：

1. 用户本轮明确要求
2. 当前会话已确认内容
3. 项目级 `EXTEND.md`
4. 用户级 `EXTEND.md`
5. `SKILL.md` 内置默认值

对这个 skill 来说，`EXTEND.md` 是非阻塞扩展层，不是强制初始化步骤。

- 找到 `EXTEND.md`：读取、摘要、应用到本轮默认值。
- 没找到 `EXTEND.md`：直接继续执行，使用内置默认值。

`EXTEND.md` 第一版至少支持以下能力：

- 默认平台
- 默认目标读者
- 默认写作风格
- 默认目标动作
- 全局写作规则
- 扩展骨架
- 骨架触发规则
- 覆盖规则

其中“全局写作规则”不是某个骨架的附属说明，而是整个 skill 在用户未明确指定表达方式时的默认回退策略。

它至少会影响以下环节：

- 第 1 步：锁定写作简报时，推断默认语气、节奏、讲解方式
- 第 2 步：确定文章骨架时，影响推荐哪种骨架更适合当前项目偏好
- 第 6 步：整合主稿时，影响句式、解释深度、情绪强度和教学感
- 第 7 步：适配平台时，影响是否优先按口播节奏、低门槛表达和步骤化拆解来改写

如果项目里定义了扩展骨架，执行时必须优先把扩展骨架与内置骨架一起纳入候选，而不是忽略项目配置。

先判断当前任务属于哪一类，再决定读哪些参考文件。

需要稳定输出格式时，额外读取 `references/llm-prompts.md`，直接复用其中对应任务的提示词模板。

## 核心模型

始终把内容视为四层叠加，而不是只修“结构”。

1. 结构层：写清楚。处理结论先行、分组、顺序、删冗余。
2. 叙事层：写好看。处理张力、场景、人物、节奏、转折。
3. 说服层：写可信。处理证据、机制、反方、边界、行动理由。
4. 传播层：写得出去。处理标题、钩子、记忆点、情绪点、分享动机。

输出前检查四层是否都覆盖。缺一层，内容就容易出现以下问题：

- 只有结构，没有可读性：像说明书。
- 只有叙事，没有判断：像流水账。
- 只有立场，没有证据：像情绪宣泄。
- 只有标题，没有正文支撑：像低质诱导。

## 工作流

以下流程必须按“提问/确认 -> 生成文件 -> 请求确认 -> 进入下一步”的顺序执行。

## 交互提问模板

以下提问模板是默认执行骨架。可以根据上下文删减，但不能跳过“提问 -> 等待回答 -> 记录确认”的基本结构。

### 通用回合模板

每一轮回复尽量包含这 4 个部分：

1. 当前步骤：例如“当前是 Step 2，正在确认文章骨架”。
2. 本步产出：说明刚生成或将生成哪个文件。
3. 本轮问题：只问最关键的 1 到 3 个问题。
4. 确认提示：明确告诉用户“你确认后我继续下一步”。

推荐句式：

- 我现在在第 X 步，先确认这一层，不直接往后写。
- 本步会生成 `某个文件`，先把你最关键的判断锁定。
- 你回复这 1 到 3 个问题后，我再继续下一步。

### 第 0 步提问模板

适用：刚进入任务、信息还不完整。

推荐提问：

- 这次你想做的是从零写、重写、诊断，还是多平台改编？
- 这篇内容你优先想发到哪里？如果没想好，我先按长文推进。
- 这次最想表达的主题，先用一句人话说出来是什么？

### 第 1 步提问模板

适用：确认简报。

推荐提问：

- 这篇内容你最想写给谁看？
- 读完之后，你最希望对方记住哪一个判断？
- 你手上已经有的素材是什么：经历、案例、研究、旧稿，还是只有一个主题？

如果信息不足，可以补一句：

- 我先按现有信息补一个最小简报，你看哪里不对我再改。

### 第 2 步提问模板

适用：确认文章骨架。

推荐提问：

- 我这里有 2 到 3 种骨架，你更想走“观点直给”“误区重构”还是“故事带出判断”？
- 如果你不想选，我可以按当前素材替你定一个最合适的骨架。

### 第 3 步提问模板

适用：确认叙事抓手。

推荐提问：

- 这篇里你最想让读者先看到哪个场景？
- 真正的冲突是什么：努力没回报、认知错位，还是结果和预期相反？
- 你有没有一个具体瞬间、对话、动作或数字，能让内容立刻落地？

### 第 4 步提问模板

适用：确认说服材料。

推荐提问：

- 你最能拿来支撑这个观点的证据是什么？
- 如果有人反对你，他最可能怎么反驳？
- 这条判断在什么情况下不成立，或者代价是什么？

### 第 5 步提问模板

适用：确认传播方向。

推荐提问：

- 你更想让这篇内容靠“反常识”“强代价”还是“强共鸣”打开？
- 这篇更适合被收藏、转发，还是被评论讨论？
- 如果只能留一句最容易被复述的话，你希望是哪句？

### 第 6 步提问模板

适用：确认整合稿方向。

推荐提问：

- 主稿我已经整合出来了，你先看大方向是否对，不急着抠字句。
- 你现在最想改的是开头、结构、语气，还是结尾判断？

### 第 7 步提问模板

适用：确认平台改编。

推荐提问：

- 接下来你要我把它改成哪些平台版本？
- 你是要保留原判断不动，只改表达形态，还是连传播角度一起调整？

### 第 8 步提问模板

适用：最终 QA 和收尾。

推荐提问：

- 这一版你是准备直接发，还是还要再做一轮精修？
- 如果只再改一处，你最希望我动哪里？
- 这轮确认后，我就把状态记为完成。

### 第 0 步：初始化会话

先创建本次写作会话，不要直接进入正文。

要做的事：

- 检查是否存在 `.daqi-skills/media-writing/EXTEND.md` 和 `$HOME/.daqi-skills/media-writing/EXTEND.md`。
- 如果存在，读取并总结其中的默认值、全局写作规则和扩展骨架。
- 判断任务类型。
- 生成日期目录 `posts/YYMMDD/{slug}/`。
- 写入 `00-会话.md`，记录主题、任务类型、目标平台、已加载扩展配置、已生效全局写作规则、当前已知素材、待确认项、下一步。
- 把本次流程拆成清晰步骤，让用户知道现在进行到哪一步。

优先使用 `assets/session-templates/00-会话-template.md` 生成 `00-会话.md`。

输出要求：

- 先告诉用户你理解到的任务类型和主题。
- 如果找到了 `EXTEND.md`，先用 2 到 4 行概括本轮生效的默认值、全局写作规则和扩展骨架。
- 再给一个最小会话摘要。
- 然后明确提问本轮最关键的 1 到 3 个问题。
- 等用户回复后，再更新或新建下一步文件。

### 第 1 步：锁定写作简报

这一轮只解决“写什么、写给谁、为什么写”。

优先提问最关键、最影响后续结构的 1 到 3 项；已有信息足够时，可以少问，但仍要把推断结果给用户确认。

- 目标平台：公众号、小红书、短视频口播、朋友圈、知乎回答等。
- 目标读者：新手、从业者、老板、创作者、泛用户。
- 核心承诺：读完能获得什么判断、方法、情绪价值或行动指令。
- 作者身份：旁观者、实操者、过来人、研究者。
- 内容素材：案例、经历、数据、观察、原始笔记、旧稿。
- 目标动作：收藏、转发、评论、私信、关注、成交。

- 如果用户只给一个主题，先基于主题生成一个最小简报，但必须把推断项标清并请求确认，不要把推断当成既定事实。

直接使用 `assets/session-templates/01-简报-template.md` 作为简报模板。
做法不是机械填空，而是先阅读用户给的主题、素材和目标，再由 LLM 推断缺失项，补成一份最小可写简报。

优先使用 `assets/session-templates/01-简报-template.md` 生成 `01-简报.md`。

文件要求：

- 生成 `01-简报.md`。
- 文件内必须区分“用户明确提供”和“模型推断待确认”。
- 结尾必须有“待用户确认项”和“下一步进入哪一类骨架选择”。

### 第 2 步：确定文章骨架

先确定文章骨架，再写句子。一次给出 1 到 3 个可选骨架，不要混乱切换，并要求用户明确选择或授权你代选。

如果 `EXTEND.md` 里定义了扩展骨架，必须与内置骨架一起评估；当触发规则命中时，优先把对应扩展骨架放进候选方案。

- 结论 -> 原因 -> 证据 -> 动作：适合观点文、建议文。
- 问题 -> 误区 -> 重构 -> 方法：适合认知升级、反常识内容。
- 现象 -> 机制 -> 推演 -> 机会/风险：适合趋势分析。
- 故事 -> 冲突 -> 反转 -> 提炼：适合个人经历、案例文。
- 清单 -> 分类 -> 示例 -> 使用边界：适合工具型、教程型内容。

具体拆解规则和论证方式见 `references/structure-and-argument.md`。

优先使用 `assets/session-templates/02-骨架-template.md` 生成 `02-骨架.md`。

文件要求：

- 生成 `02-骨架.md`。
- 至少包含：候选骨架、扩展骨架命中情况、推荐骨架、推荐理由、章节顺序、每章的小结论。
- 写完后只推进到“等待用户确认骨架”，不要直接扩写正文。

### 第 3 步：补充叙事张力

哪怕是知识内容，也至少补一个叙事抓手，不要把正文写成干瘪提纲。

这一轮要先让用户确认叙事抓手，至少确认下列内容中的两项：场景、冲突、变化、细节。

优先补这几种元素：

- 一个具体场景：让读者“看见”问题，而不是只“知道”问题。
- 一个冲突：旧认知和新认知，表象和机制，努力和结果之间的张力。
- 一个变化：人物、判断、处境或结果发生了什么改变。
- 一个细节：时间、动作、语言、数字、反应。

具体叙事方法和表达节奏见 `references/narrative-and-expression.md`。

优先使用 `assets/session-templates/03-叙事-template.md` 生成 `03-叙事.md`。

文件要求：

- 生成 `03-叙事.md`。
- 文件中要有“候选开场场景”“核心冲突”“关键变化”“可复用细节”。
- 如果用户没有素材，可以先生成候选方向，但必须标明哪些是待用户补充。

### 第 4 步：建立说服力

任何观点都至少补齐以下四件事中的三件，并要求用户确认其中最重要的一到两项是否成立：

- 证据：案例、数据、经验、对比。
- 机制：为什么会这样，而不是只说“就是这样”。
- 反方：常见质疑、反例、误解。
- 边界：适用条件、失效条件、代价。

不要把“绝对化判断”当成说服。优先追求“有边界的强观点”。

具体方法见 `references/persuasion-and-trust.md`。

优先使用 `assets/session-templates/04-说服-template.md` 生成 `04-说服.md`。

文件要求：

- 生成 `04-说服.md`。
- 文件中至少写清：主张、证据、机制、反方、边界、风险点。
- 如果证据不足，要明确写“证据待补”，不要假装已经论证完成。

### 第 5 步：设计传播点

完成正文框架后，再单独设计传播点，不要只改标题。

至少生成以下内容：

- 3 到 5 个标题，覆盖不同传播角度。
- 1 个开头钩子，确保前 3 句有张力。
- 1 个可复述金句，方便截图或引用。
- 1 个转发动机，回答“为什么别人愿意转给朋友”。

传播规则与标题机制见 `references/titles-and-distribution.md`。
标题不要靠模板硬套。先提炼正文的核心判断、目标读者、利益、反差、代价，再生成多组方向。

优先使用 `assets/session-templates/05-传播-template.md` 生成 `05-传播.md`。

文件要求：

- 生成 `05-传播.md`。
- 至少包含 3 到 5 个标题、1 个开头钩子、1 个金句、1 个转发动机。
- 用户确认传播方向后，才能进入整合主稿。

### 第 6 步：整合主稿

只有当前五步已完成确认后，才整合主稿。

整合时遵守以下要求：

- 主稿必须严格继承已确认的简报、骨架、叙事、说服和传播方向。
- 如果发现前面确认存在冲突，先暂停并指出冲突，再请用户决定。
- 成稿完成后，不要立刻改平台，先请用户确认主稿方向。

优先使用 `assets/session-templates/06-主稿-template.md` 生成 `06-主稿.md`。

文件要求：

- 生成 `06-主稿.md`。
- 文件开头附一段“本稿基于哪些已确认文件整合而成”。
- 文件结尾附一段“待确认修改点”。

### 第 7 步：适配平台

如果用户要求平台改编，把同一内容拆成“信息不变、形态变化”的版本。

- 公众号：强化结构、论证、案例展开。
- 小红书：强化开头代入、清单感、情绪标签、可收藏性。
- 口播稿：强化短句、节奏、停顿、复述点。
- 短帖：保留一个核心判断和一个最强例子。

改编时不要机械缩写。先保留核心承诺，再调整顺序和节奏。

直接使用 `assets/session-templates/07-改编-template.md` 作为平台改编模板。
如果已经有长文草稿，先完整阅读，再由 LLM提炼“核心判断、关键例子、冲突、结尾判断”，然后按平台重写，不要做句子级裁剪。

优先使用 `assets/session-templates/07-改编-template.md` 生成 `07-改编.md`。

文件要求：

- 生成 `07-改编.md`。
- 一个文件内可以放多个平台版本，但必须按平台分节。
- 如果用户暂时不做平台改编，也要写明“本轮跳过平台改编”的决策原因。

### 第 8 步：执行最终校验

交付前做最后检查：

- 第一段是否在 30 秒内让读者知道“这篇为什么值得看”。
- 每一部分是否只服务一个小结论。
- 是否有至少一个具体例子或场景。
- 是否解释了关键机制，而不是只有口号。
- 是否存在过度承诺、绝对化表述、空泛金句。
- 标题是否与正文承诺一致，而非单纯刺激点击。
- 结尾是否给出明确判断、行动或余味。

可直接使用 `references/workflow-templates.md` 中的诊断清单。
如果用户给的是现有草稿，先由 LLM 按“结构、叙事、说服、传播”四层做诊断，再决定重写策略。诊断必须基于全文实际内容，不要依赖词频或标记词命中。

优先使用 `assets/session-templates/08-校验-template.md` 生成 `08-校验.md`。

文件要求：

- 生成 `08-校验.md`。
- 必须写清：通过项、未通过项、建议修改、用户最终确认状态。
- 如果用户确认收尾，可在该文件末尾写“本轮完成”。

## LLM 执行模式

### 简报生成

当用户信息不完整时，直接读取 `assets/session-templates/01-简报-template.md`，用现有上下文补齐以下字段，但必须把推断项显式标成“待确认”：

- 主题
- 平台
- 目标读者
- 核心判断
- 想让读者获得的结果
- 可用素材
- 优先使用的案例/经历
- 期望语气
- 目标动作

缺失项可以推断，但要避免编造具体事实。拿不准时用“待补充”或保留空白，不要假装知道。
如果希望输出更稳，使用 `references/llm-prompts.md` 中的“简报生成模板”。

### 草稿诊断

读取全文后，从以下四层分别给出判断。诊断完成后不要直接全文重写，要先让用户确认“是否按此方向重写”。

- 结构：主结论是否明确，分组是否清楚，顺序是否服务理解。
- 叙事：是否有场景、冲突、变化、细节，阅读是否有推进感。
- 说服：是否有证据、机制、反方、边界。
- 传播：标题、开头、金句、转发动机是否成立。

诊断输出必须包含三部分：

- 最值得保留的内容
- 最影响传播的问题
- 下一版最优先修改的 3 件事

需要更强约束时，使用 `references/llm-prompts.md` 中的“草稿诊断模板”。

文件要求：

- 诊断结果优先写入 `01-诊断.md`。
- 如果用户确认重写，再生成 `02-重写计划.md` 和后续稿件。
- `01-诊断.md` 优先使用 `assets/session-templates/01-诊断-template.md`。
- `02-重写计划.md` 优先使用 `assets/session-templates/02-重写计划-template.md`。

### 重写计划

在用户确认“按此方向重写”之后，再进入重写计划，不要跳过这一步直接改稿。

重写计划至少写清：

- 哪些内容保留不动。
- 哪些内容要删除、合并或重排。
- 开头准备怎么改。
- 论证链准备怎么补。
- 重写后优先追求的传播目标是什么。

文件要求：

- 重写计划写入 `02-重写计划.md`。
- 文件中必须有“保留项”“删除项”“重组方案”“本轮重写目标”。
- 用户确认计划后，才进入 `03-主稿.md`。

### 标题生成

先总结正文的：

- 一句话核心判断
- 目标读者是谁
- 读者能获得什么
- 什么地方最反常识
- 什么损失最值得强调

然后再按不同传播角度各写一组标题，不要围绕一个模板机械改词。生成后先让用户选择方向，再回写到 `05-传播.md`。
需要稳定批量出标题时，使用 `references/llm-prompts.md` 中的“标题生成模板”。

### 平台改编

使用 `assets/session-templates/07-改编-template.md` 作为输出骨架，但内容生成方式必须是重写，不是抽句拼接。改编前先确认用户要哪些平台，不要默认全部生成。

改编时遵守以下原则：

- 公众号版：保留论证链和案例展开。
- 小红书版：强化代入、清单感和收藏价值。
- 口播版：改成短句、停顿点明显、适合说出口。
- 短帖版：只保留一个判断和一个最强例子。

如果原文核心判断不清，先重提炼观点，再改编，不要带着模糊观点拆平台。
需要固定输出骨架时，使用 `references/llm-prompts.md` 中的“平台改编模板”。

## 来源框架

使用以下知识框架，不必显式提书名，但要落实其方法。

- 《金字塔原理》+《麦肯锡教我的写作武器》：结论先行、分组表达、逻辑树。
- 《逻辑思考力》+《学会提问》：MECE、论证完整性、识别偷换概念与证据不足。
- 《让创意更有黏性》：简单、意外、具体、可信、情绪、故事。
- 《疯传》+《引爆点》：分享动机、触发器、情绪、公共性、实用价值。
- 《故事》+《写作这回事》+《非虚构写作》：冲突、变化、真实、简洁、清晰。
- 《学会写作》+《爆款写作》：标题、开头、平台感、用户收益表达。

## 输出规则

根据任务类型直接交付结果，不要只给原则，但交付方式必须是交互式逐步推进。

- 写作任务：按步骤提问和确认，最终再给成稿。
- 重写任务：先诊断并确认重写方向，再重写，再用 3 到 5 行说明改动逻辑。
- 诊断任务：先指出问题，等待用户确认后，再给重写版或修订建议。
- 标题任务：一次性给多组方向，但仍要让用户确认采用哪组，再落盘。
- 平台改编任务：先确认目标平台，再并排给出不同版本，保留共同核心观点。

每次回复都要说明：

- 当前处于第几步。
- 本步已生成哪个文件。
- 还需要用户确认什么。

除非用户明确要求，否则不要输出大段“写作理论讲解”。理论应隐藏在产出背后。
