# 结构与论证

## 目录

- 目标
- 文章骨架
- 分组原则
- 论证链条
- 常见失误

## 目标

先让读者快速明白结论，再让他愿意继续读细节。不要把“铺垫”误当成“高级感”。

## 文章骨架

优先从这五类中选一种主骨架：

### 1. 结论 -> 原因 -> 证据 -> 动作

适合建议文、方法文、判断文。

- 开头直接给核心判断。
- 中段拆 2 到 4 个原因。
- 每个原因补证据或案例。
- 结尾给行动建议或适用边界。

### 2. 问题 -> 误区 -> 重构 -> 方法

适合反常识、认知升级内容。

- 先指出读者熟悉的问题。
- 再指出大众常见误判。
- 再给新框架。
- 最后给能落地的方法。

### 3. 现象 -> 机制 -> 推演 -> 决策

适合行业分析、趋势判断。

- 现象不要堆砌新闻。
- 机制要回答“为什么”。
- 推演要交代条件。
- 决策要说明谁该做什么。

### 4. 故事 -> 冲突 -> 反转 -> 提炼

适合个人成长、案例拆解。

- 让故事只承担“引入”和“证明”，不要喧宾夺主。
- 冲突必须清晰：想要什么、卡在哪里、付出什么代价。
- 提炼阶段回到普遍意义。

### 5. 清单 -> 分类 -> 示例 -> 边界

适合工具推荐、经验总结。

- 分类维度必须单一。
- 每一点至少配一个例子。
- 最后补“什么情况下别这么做”。

## 分组原则

使用接近 MECE 的分组方式，但不要为了形式牺牲可读性。

- 每一组只回答一个问题。
- 同级标题保持同一语法结构。
- 相邻段落避免重复同一意思。
- 先总后分，分完要回收。

一个好用的检验法：把二级标题单独拎出来看，读者是否仍能理解全文骨架。

## 论证链条

每个核心判断尽量具备这条链：

主张 -> 依据 -> 机制 -> 反例/异议 -> 结论边界

可以缩写，但不要长期缺位。尤其要避免以下偷懒：

- 只有主张，没有依据。
- 只有案例，没有机制。
- 只有数据，没有解释。
- 只有立场，没有边界。

## 常见失误

- 开头绕太久，读者 10 秒内仍不知道要说什么。
- 一段里塞多个结论，导致读者抓不住重点。
- 小标题只是修辞，不承担信息功能。
- 例子和结论脱节，像贴素材。
- 结尾重复开头，没有推进到行动或判断。
